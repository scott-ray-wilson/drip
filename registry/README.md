# registry

The element catalog: the single source of truth for what UI elements drip
offers, how they're described and categorized, and how they depend on one
another. Everything downstream reads from here: [`codegen`](../codegen) turns it
into the release `dist/` and the [docs site](https://drip.pink), while the
elements it lists actually live in [`ui`](../ui).

`drip_registry` is an **internal package**. It is never published to Hex (see
[`PUBLISHING.md`](../PUBLISHING.md)); it exists only to be compiled as a path
dependency inside this repo.

## What's here

A single module, `drip/registry`, that is **pure data with no
platform-specific code**. `codegen` (which runs on Erlang) and `docs` (which runs
on JavaScript) both compile it, so it must stay target-agnostic.

It holds *metadata about* elements, not the elements themselves. An element's
Gleam, CSS, and FFI sources live in [`ui`](../ui); the emitted `registry.json`
index and `theme.css` are produced by `codegen`. This package just says which
elements exist and how they relate.

## The `Element` record

Each entry is one `Element`. The catalog is the `all` list of them.

| Field | Type | What it is |
|-------|------|-----------|
| `name` | `String` | Slug and file stem. The element vendors to `src/<prefix>/<name>.*` and is addressed as `drip add <name>`. |
| `description` | `String` | One-line summary shown on the docs site and in `drip list`. |
| `category` | `Category` | Which catalog section it belongs to. |
| `ffi` | `Bool` | Whether the element ships a `<name>.ffi.mjs` beside its Gleam module. |
| `registry_dependencies` | `List(String)` | Other elements this one needs, by name. |
| `dependencies` | `List(String)` | Gleam packages this element needs. |

`registry_dependencies` are resolved and
vendored automatically by the CLI, while `dependencies` are surfaced as
`gleam add` steps for the user to run.

`Category` is a closed set of eight sections: `Forms`, `Actions`, `Navigation`,
`Overlay`, `Disclosure`, `Layout`, `DataDisplay`, and `Feedback` primarily used for the [docs site](https://drip.pink).

## Queries

| Function | What it returns |
|----------|-----------------|
| `all` | Every element, in catalog order. |
| `find(registry, name)` | `Ok(Element)` for a name, or `Error(Nil)`. |

Resolving an element's **transitive dependency
closure** (the order `drip add` vendors things in) is deliberately *not* here.
Hex bars path dependencies, so the CLI can't share this code; it carries its own
leaner reader over the emitted `registry.json` and owns closure resolution alone.

## Adding or editing an element

Add a `pub const <name>: Element = Element(...)` and append it to the `all` list.
That is the catalog side; the element's actual files land in [`ui`](../ui) and its
docs page and snippets are generated by `codegen`. An element that isn't ready to
ship simply has no entry here: its `ui/` sources build in-repo without reaching
the emitted registry, the docs, or `drip add`.

## Usage

```sh
gleam test   # source-data integrity checks
```
